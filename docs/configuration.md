# Configuration reference

`cmdpeek` reads a YAML catalog containing reusable shell commands and their interactive variables.

## Configuration lookup

`cmdpeek` resolves the configuration path in this order:

```text
1. --config <path>
2. CMD_PEEK_CONFIG_FILE
3. .cmdpeek.yaml
```

Examples:

```bash
cmdpeek --config examples/basic.yaml
```

```bash
export CMD_PEEK_CONFIG_FILE="$HOME/.config/cmdpeek/commands.yaml"
cmdpeek
```

## Minimal configuration

```yaml
version: 1
shell: bash

commands:
  - name: hello
    title: Say hello
    description: Print a greeting
    labels:
      - example
      - greeting
    run: echo "Hello from cmdpeek"
```

## Top-level fields

| Field | Required | Description |
|---|---:|---|
| `version` | No | Configuration format version. Defaults to `1`. |
| `shell` | No | Default shell used for commands. Defaults to `sh`. |
| `commands` | Yes | Non-empty list of command definitions. |

Only configuration version `1` is currently supported.

## Command fields

```yaml
commands:
  - name: github-run-logs
    title: Download GitHub Actions run logs
    description: Download and extract logs from a workflow run
    labels:
      - github
      - actions
      - logs
    shell: bash
    run: |
      set -euo pipefail
      echo "{{url}}"
    variables:
      - name: url
        prompt: Workflow run URL
        source:
          type: input
```

| Field | Required | Description |
|---|---:|---|
| `name` | Yes | Unique internal identifier. |
| `title` | No | Human-readable title. Defaults to `name`. |
| `description` | No | Explanation shown in search and detail views. |
| `labels` | No | Searchable categories, aliases and alternate vocabulary. |
| `shell` | No | Overrides the top-level shell for this command. |
| `run` | Yes | Shell command or multiline script. |
| `variables` | No | Interactive values referenced by the command. |

Command names must be unique.

## Variables

Variables are referenced using:

```text
{{variable_name}}
```

Example:

```yaml
run: kubectl get pods -n "{{namespace}}"
```

Every referenced value must be declared in `variables`:

```yaml
variables:
  - name: namespace
    prompt: Select namespace
    description: Namespace from the current Kubernetes context
    source:
      type: command
      command: >-
        kubectl get namespaces
        -o jsonpath='{range .items[*]}{.metadata.name}{"\n"}{end}'
```

## Variable fields

| Field | Required | Description |
|---|---:|---|
| `name` | Yes | Variable identifier used in `{{...}}`. |
| `prompt` | No | Text displayed to the user. Defaults to `name`. |
| `description` | No | Additional explanation. |
| `default` | No | Initial or preselected value. |
| `source` | Yes | Defines how the value is obtained. |

Supported source types:

```text
input
options
environment
command
```

## Multiline commands

Use YAML block syntax for scripts:

```yaml
run: |
  set -euo pipefail

  output="{{output}}"
  mkdir -p "$(dirname "$output")"
  printf '%s\n' "Generated by cmdpeek" > "$output"

  echo "Saved to: $output"
```

The configured shell executes the rendered script using:

```text
<shell> -c <rendered command>
```

## Shell selection

Global shell:

```yaml
version: 1
shell: bash
```

Per-command override:

```yaml
- name: portable-example
  shell: sh
  run: |
    value="{{value}}"
    printf '%s\n' "$value"
```

Use `bash` when scripts depend on features such as:

```bash
set -o pipefail
[[ ... ]]
BASH_REMATCH
arrays
```

## Labels

Labels are first-class searchable metadata.

```yaml
labels:
  - kubernetes
  - pods
  - logs
  - troubleshooting
  - stream
  - klo
```

They can represent:

- technologies;
- resources;
- actions;
- teams or projects;
- historical aliases;
- alternative vocabulary.

Prefer several focused labels over one long label.

## Validation rules

The loader rejects:

- missing command names;
- duplicate command names;
- missing `run` values;
- duplicate variable names;
- unsupported variable source types;
- empty `options` lists;
- empty option values;
- `environment` sources without a variable name;
- `command` sources without a command;
- command sources referencing variables declared later.

## Complete example

```yaml
version: 1
shell: bash

commands:
  - name: pod-logs
    title: Follow Kubernetes pod logs
    description: Select a namespace and stream logs from one of its pods
    labels:
      - kubernetes
      - pods
      - logs
      - troubleshooting
      - klo
    run: kubectl logs -f "{{pod}}" -n "{{namespace}}"
    variables:
      - name: namespace
        prompt: Select namespace
        description: Namespace from the current Kubernetes context
        source:
          type: command
          command: >-
            kubectl get namespaces
            -o jsonpath='{range .items[*]}{.metadata.name}{"\n"}{end}'

      - name: pod
        prompt: Select pod
        description: Pod discovered inside the selected namespace
        source:
          type: command
          command: >-
            kubectl get pods -n "{{namespace}}"
            -o jsonpath='{range .items[*]}{.metadata.name}{"\n"}{end}'
```

## Testing a configuration

Run the catalog directly:

```bash
cmdpeek --config path/to/commands.yaml
```

For repository-local catalogs:

```bash
cp examples/basic.yaml .cmdpeek.yaml
cmdpeek
```
